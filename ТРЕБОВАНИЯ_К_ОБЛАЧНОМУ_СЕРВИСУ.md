# 🌐 ТРЕБОВАНИЯ К ОБЛАЧНОМУ СЕРВИСУ ДЛЯ ANSWERS PLATFORM

## 📊 ТЕКУЩАЯ АРХИТЕКТУРА ПРОЕКТА

### Технологический стек:
- **Backend:** Python 3.14+, FastAPI, SQLAlchemy
- **Frontend:** Next.js 14, React 18, TypeScript, Tailwind CSS
- **База данных:** PostgreSQL 15 с расширением pgvector (для векторного поиска)
- **Кэширование:** Redis 7
- **ML/AI:** Sentence Transformers для эмбеддингов, интеграция с LLM провайдерами
- **Контент:** 80 HTML-статей на 4 языках (~260,000 слов)
- **Контейнеризация:** Docker Compose

## 🔧 СПЕЦИФИЧЕСКИЕ ТРЕБОВАНИЯ ПО КОМПОНЕНТАМ

### 1. PostgreSQL с pgvector

**Критически важные настройки:**
```sql
-- Минимальная конфигурация
shared_buffers = 25% RAM
effective_cache_size = 75% RAM
work_mem = 64MB - 256MB
maintenance_work_mem = 512MB - 1GB

-- Для pgvector оптимизации
max_parallel_workers_per_gather = 2-4
parallel_tuple_cost = 0.01
```

**Storage требования:**
- Каждый векторный эмбеддинг: ~768 float32 values = 3 KB
- 80 статей × среднее 50 чанков = 4,000 векторов
- Общий размер векторной базы: ~12 MB (минимально)
- С ростом контента (1000+ статей): 1-5 GB

**Backup стратегия:**
- Ежедневные полные бэкапы
- WAL archiving для point-in-time recovery
- Retention: 7-30 дней
- Off-site backup replication

### 3. Backend (FastAPI + Python)

**Зависимости требующие ресурсов:**
```python
# Resource-intensive libraries:
- sentence-transformers: Требует 2-4 GB RAM для загрузки моделей
- langchain: Moderate CPU usage
- asyncpg: Low overhead
- pydantic: Minimal resources
```

**Оптимизация:**
- Использовать Gunicorn с Uvicorn workers
- Workers: `(2 × CPU_cores) + 1`
- Пример для 4 vCPU: 9 workers
- Worker connections: 1000-2000

**Environment variables:**
```bash
DATABASE_URL=postgresql://user:pass@host:5432/db
REDIS_URL=redis://host:6379
GROQ_API_KEY=***
OPENROUTER_API_KEY=***
EMBEDDING_MODEL=sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2
```

## 📞 ПОДДЕРЖКА И ОПТИМИЗАЦИЯ ЗАТРАТ

### Cost Optimization Tips:

1. **Использовать Spot Instances** (AWS) или preemptible VMs (GCP) для non-critical workloads
2. **Reserved Instances** для predictable workload (экономия 30-40%)
3. **Auto-scaling** чтобы не платить за неиспользуемые ресурсы
4. **CDN caching** для снижения нагрузки на backend
5. **Database query optimization** для снижения CPU usage
6. **Efficient caching** стратегии для снижения LLM API calls
7. **Monitor and right-size** регулярно пересматривать resource usage

### Support Options:

**Self-managed:**
- Pros: Полный контроль, ниже стоимость
- Cons: Требует DevOps expertise, 24/7 responsibility

**Managed Services:**
- Pros: Автоматические обновления, backups, monitoring
- Cons: Higher cost, less flexibility

**Hybrid Approach (Recommended):**
- Managed database (PostgreSQL)
- Managed cache (Redis)
- Self-managed application servers
- Это дает баланс контроля и удобства
